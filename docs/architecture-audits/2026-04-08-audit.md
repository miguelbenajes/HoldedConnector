# Architecture Audit Report — holded-connector

**Date:** 2026-04-08
**Overall Health Score:** 3.83/10

## Executive Summary

The holded-connector is a functional but structurally stressed Python FastAPI service. The three core files (connector.py at 2,825 lines, api.py at 2,222 lines, ai_agent.py at 1,645 lines) contain the vast majority of code with minimal separation of concerns. There are 76 endpoints in a single api.py file, massive copy-pasted item-building logic across 4 locations, and critically, several /api/agent/* endpoints bypass the Write Gateway's 6-stage validation pipeline. The write gateway itself is a bright spot architecturally, but is undermined by these bypasses.

## Critical Issues

### C1. Production secrets in shared filesystem
- `.env` and `.env.production` contain real API keys, DB passwords, service keys
- Present in `IA SHARED/` directory accessible to other contexts
- `.gitignore` covers them, but filesystem exposure remains

### C2. Dual write paths bypass the safety gateway
- `/api/agent/invoice` (api.py:1512) calls `connector.create_invoice()` directly
- Same for `/api/agent/estimate` (1576), `/api/agent/contact` (1653), `/api/agent/invoice/{id}/status` (1792), `/api/agent/send` (1807)
- These skip the Write Gateway's validation, rate limiting, preview, and audit logging
- The AI agent's write tools correctly route through `gateway.execute()`

## High Issues

### H1. God files — all three core files exceed 1,500 lines
- connector.py: 2,825 lines — DB, Holded sync, schema, amortization, analysis, audit
- api.py: 2,222 lines, 76 endpoints — routes + business logic + SQL
- ai_agent.py: 1,645 lines — tool defs, executors, prompts, streaming, history

### H2. Copy-pasted item-building logic across 4 locations
- api.py `agent_create_invoice()` (1516-1541)
- api.py `agent_create_estimate()` (1580-1604)
- api.py `agent_update_estimate()` (1621-1643)
- write_gateway.py `_build_holded_payload()` (152-181)
- Differ subtly in taxes, retention, account, desc handling

### H3. HTTPException imported inline 12 times in api.py

### H4. No separation between route layer and data access
- Job endpoints, audit endpoints contain raw SQL in route handlers
- Mix of `get_db_connection()` context manager vs raw `connector.get_db()`/`release_db()`

## Medium Issues

### M1. Inconsistent DB access patterns
- Context manager (`_ConnProxy`) vs raw get/release — missed finally can leak connections

### M2. Reports file listing duplicated (ai_agent.py + api.py)

### M3. Rate limiting implemented twice independently
- ai_agent.py: dict-based `{ip: [timestamps]}`
- write_gateway.py: `RateLimiter` class with sliding window

### M4. Test coverage minimal — 4 test files, only write_gateway covered

### M5. Tool definitions as raw dicts, no Pydantic validation

### M6. CORS `*` in production (`ALLOWED_ORIGINS=*`)

### M7. _CompatCursor/_ConnProxy wrappers duplicate connector functionality
